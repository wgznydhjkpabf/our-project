## 01 项目概述与背景

本章节深入探讨项目的研究背景与现实意义，明确项目建设的核心目标与设计初衷，为系统开发奠定理论基础。

### 一、研究背景：直击校园交易核心痛点

1. **闲置交易需求旺盛**  
   校园内教材、电子产品、生活用品等闲置资源存量充足，学生群体对高性价比二手商品存在长期、强烈的消费需求。

2. **交易信息高度分散**  
   学生闲置交易主要依靠微信群、校园论坛等渠道，商品信息杂乱无章，检索筛选难度大，整体交易效率极低。

3. **交易信任机制缺失**  
   现有渠道无校园实名身份校验、无买卖双方评价体系，虚假商品、交易欺诈、财物纠纷问题频发。

4. **交易流程混乱无标准**  
   线上议价、线下交付无统一流程规范，买卖双方沟通成本高，极易产生沟通分歧与交易误解。

### 二、项目意义：构建安全高效的校园二手交易新生态

1. **提升校园闲置交易效率**  
   统一整合二手商品发布渠道，搭配商品智能筛选、分类检索功能，快速匹配供需双方，大幅降低信息查找与沟通成本。

2. **搭建交易安全保障体系**  
   搭建校园实名认证、买卖双向信誉互评、平台全程交易监管三重机制，遏制虚假交易与欺诈行为，全方位保障学生用户合法权益。

3. **促进校园闲置资源循环利用**  
   盘活学生闲置物品，延长商品使用周期，减少资源浪费，践行绿色低碳、可持续发展的生活理念。

4. **团队开发能力全面进阶**  
   完整落地软件工程全流程真实项目开发，实操主流前后端开发技术，锻炼团队协作、需求拆解、问题排查与工程落地能力。

### 三、项目核心目标

打造集闲置信息整合、交易安全保障、校园资源循环利用于一体的专属校园二手交易服务平台。

### 四、总体开发目标

1. **基础功能目标**：设计并实现界面友好、功能完善、安全稳定的校园二手交易平台，支持学生发布、浏览、购买二手商品，同时为平台管理员提供商品审核、用户管理、数据统计等后台管理功能，针对性解决校园线下/社群二手交易现存各类痛点。

2. **工程流程目标**：严格遵循软件工程标准流程，完整走完需求分析→系统设计→编码开发→功能测试→项目部署全流程，建立标准化、规范化项目开发体系。

3. **数据架构目标**：采用前后端分离架构，数据库设计满足第三范式3NF，熟练运用MySQL存储过程、数据库触发器，搭建稳定高效的数据持久层。

4. **团队能力目标**：熟练掌握Web应用开发全栈技术，使用Git工具统一管理项目代码版本，通过多人协同开发交付规范、高质量代码，培养标准化软件工程开发素养。

## 02 系统设计与技术方案

本章节深度解析项目技术架构、功能模块划分、核心业务流程，搭建高稳定、可扩展的系统底层支撑，为业务功能落地提供技术保障。

### 一、系统整体设计原则

整体遵循模块化开发、高内聚低耦合设计准则，将业务逻辑层与前端交互、数据存储分层隔离，架构清晰、便于后期迭代扩展，保障系统长期可维护性。

### 二、整体技术分层架构（B/S架构+前后端分离）

整体采用经典B/S浏览器服务端架构，严格执行前后端分离开发模式，以RESTful JSON API作为前后端统一数据交互标准，实现各层级解耦，提升并行开发、后期维护效率。

1. **前端交互层**  
   核心框架使用React，搭配Ant Design组件库搭建页面，Axios库处理网络请求，实现响应式、流畅直观的用户操作界面。

2. **后端服务层**  
   基于Spring Boot 3开发，整合MyBatis完成数据持久化操作，使用JWT技术实现无状态登录身份认证，提供稳定、安全的业务逻辑处理能力。

3. **数据存储层**  
   数据库选用MySQL 8.0，依托其高性能、高可靠特性，通过合理数据表结构设计、索引优化，支撑平台海量商品、订单、用户数据快速存储与检索。

### 三、系统全部功能模块划分

校园二手交易平台共分为七大核心子系统：

1. **用户子系统**：用户注册、登录、个人资料修改、收货地址管理
2. **商品子系统**：商品发布、编辑、浏览、搜索、下架、管理员商品审核
3. **订单子系统**：创建订单、订单确认、订单完成、订单取消、订单状态流转管理
4. **评价子系统**：订单星级评分、文字评论、自动计算更新卖家信誉分
5. **消息子系统**：买卖双方站内私信、平台系统通知推送
6. **地址子系统**：用户收货地址新增、修改、删除、默认地址设置
7. **管理子系统**：用户账号管理、商品审核、商品分类配置、交易数据统计分析

### 四、六大子系统详细功能说明

1. **用户子系统**  
   覆盖用户全生命周期管理，包含账号注册登录、校园身份实名认证、个人基础信息维护、多收货地址增删改查等基础功能。

2. **商品子系统**  
   支撑二手商品全生命周期管理，支持商品发布、信息编辑、关键词/分类智能搜索、商品详情展示；内置管理员商品审核机制，过滤违规虚假商品，保障平台商品合规真实。

3. **交易子系统（平台核心模块）**  
   统筹全部交易业务，负责订单创建、订单状态流转、线下面交/邮寄两种履约模式、交易结算闭环，全程管控每一笔交易流程，保障交易安全可控。

4. **评价子系统**  
   搭建买卖双向评价体系，买家可对已完成订单进行星级打分与文字评论；系统根据历史评价自动计算、展示卖家信誉分数，构建平台交易信用基础。

5. **消息子系统**  
   集成点对点站内私信、平台系统通知两大功能，支持消息实时推送，保障买卖双方咨询沟通顺畅，交易关键节点自动推送提醒消息。

6. **管理子系统**  
   面向平台运营管理员，提供用户权限管控、商品分类目录维护、全平台交易数据统计可视化功能，支撑平台日常运营与管理决策。

### 五、系统核心业务完整流程（闭环流程）

**完整链路**：卖家发布商品 → 管理员后台审核商品信息 → 买家浏览商品并下单 → 卖家确认订单 → 买卖双方选择面交/邮寄完成实物交付 → 买家确认交易完成 → 买家发布订单评价 → 系统自动更新卖家信誉分 → 商品标记为已售，交易数据归档保存

全流程每一步操作均记录状态，数据全程可追溯，权责划分清晰可控。

1. **发布与准入审核环节**：卖家提交商品信息后，管理员核验商品真实性、合规性，审核通过商品才可前台展示上架，把控平台商品准入质量。
2. **下单与履约交付环节**：买家提交订单，卖家确认订单有效性；双方自主选择线下校园面交或线上邮寄两种交付方式，完成实物交割。
3. **评价与信誉更新环节**：交易全部完成后买家可发布评价，系统自动更新卖家信誉分数，建立正向信用激励机制。
4. **数据归档流转环节**：完成交易的商品自动标记“已售”，全部订单、评价数据持久化存入数据库，完整留存交易全链路记录，支持随时回溯查询。

### 六、数据库整体设计方案

1. **设计规范**：严格遵循数据库第三范式3NF，消除数据冗余、更新/插入/删除异常，保证数据结构稳定；设置主键、外键、CHECK约束，配套存储过程、数据库触发器保障数据一致性。

2. **7张核心数据表及说明**  
   - `sys_user` 用户表：存储用户账号信息，学号、邮箱字段设置唯一约束，用于校园实名校验  
   - `goods_category` 商品分类表：维护平台全部商品类目，实现商品分类检索  
   - `goods` 商品表：存储二手商品信息，外键关联用户表、分类表，价格字段设置CHECK约束（价格必须大于0）  
   - `trade_order` 订单表：存储交易订单数据，外键关联商品、买家、卖家用户信息  
   - `order_review` 评价表：存储订单评价内容，一个订单仅对应一条评价，评分限制1~5星  
   - `user_message` 站内私信表：记录用户之间、平台推送的全部消息内容与收发状态  
   - `user_address` 收货地址表：存储用户多组收货地址信息

3. **数据库高级特性设计**  
   - **存储过程 `sp_complete_order`**  
     功能：订单确认完成时自动触发执行，将对应商品状态更新为“已售”，同时根据订单评价重新计算卖家信誉分。  
     设计目的：封装复杂交易业务逻辑，依托数据库原子操作保证多表数据同步更新一致性，减少后端业务代码冗余。  
   - **触发器 `trg_order_status_before_update`**  
     功能：在订单状态执行更新操作前自动校验，禁止将“已完成”“已取消”的订单修改为其他非法状态。  
     设计目的：在数据库底层强制约束业务规则，从根源杜绝人为、代码层面非法篡改订单数据，保障核心交易数据完整准确。

### 七、各模块详细功能任务清单

1. **用户管理功能**  
   - 学号+邮箱双重唯一校验注册、JWT无状态登录认证  
   - 个人信息修改、收货地址增删改查管理

2. **商品管理功能**  
   - 商品发布、编辑、下架（支持填写标题、分类、价格、成色、上传多张图片）  
   - 商品分页列表展示、分类筛选、关键词模糊搜索  
   - 管理员后台商品审核操作

3. **订单交易功能**  
   - 创建订单，支持面交、邮寄两种交易模式  
   - 订单四状态流转：待确认→待交易→已完成/已取消  
   - 交易确认完成后，系统自动将商品标记为已售

4. **评价信誉功能**  
   - 1~5星级打分+文字评论，单个订单仅支持一条评价  
   - 根据历史评价动态计算、前台展示卖家信誉分数

## 03 项目实施与计划

本章节介绍项目技术选型依据、团队分工职责、分阶段开发流程，展示当前项目阶段性开发成果与进度。

### 一、技术选型与开发规范

1. **后端核心技术**：Java语言，Spring Boot 3 + MyBatis框架，保障系统稳定、可扩展
2. **前端界面技术**：React框架搭配Ant Design组件库，开发响应式现代化交互页面
3. **数据与版本管理**：MySQL 8.0存储业务数据；Git工具统一管理项目代码版本，规范多人协作
4. **开发产出标准**：严格管控代码质量，团队每人完成核心业务代码不少于600行，保证平台功能完整可用

### 二、团队分工与协作规划

团队共4人，采用模块化并行开发分工策略：组长统筹整体项目架构、协调各模块对接；其余成员分别负责商品、数据库、后端核心逻辑、前端页面开发，同步推进开发，提升整体效率。

- **成员A**：商品模块CRUD、商品搜索功能、商品图片上传功能开发
- **成员B**：整体数据库设计、存储过程与触发器编写、图片上传辅助开发
- **成员C**：后端项目框架搭建、JWT登录鉴权、订单状态机逻辑、消息与评价核心模块开发
- **成员D**：前端页面布局开发、前后端接口联调、全系统功能测试

### 三、项目目录结构（前后端分离）

项目根目录命名 `campus-trade`，分为 `backend` 后端、`frontend` 前端两大独立模块，配套SQL脚本、项目说明文档，实现视图层与业务逻辑层完全解耦：
campus-trade/
├── backend/ (Spring Boot Java后端)
│ ├── controller/ # 接口控制层
│ ├── service/ # 业务逻辑层
│ ├── mapper/ # 数据访问层
│ └── entity/ # 数据库实体类
├── frontend/ (React前端)
│ ├── pages/ # 系统页面文件
│ ├── components/ # 公共复用组件
│ └── api/ # 后端接口请求封装
└── sql/ # 存放全部数据库建表语句、存储过程、触发器脚本

### 四、分模块开发内容规划

1. **用户模块**：基于JWT实现安全登录认证，完整实现个人资料、收货地址管理全流程
2. **商品模块**：覆盖商品发布、搜索、详情查看、下架全生命周期，集成图片上传与展示功能
3. **订单模块**：完整实现下单、履约、交易完成、评价闭环，实时追踪订单状态变更
4. **消息互动模块**：搭建站内私信通讯系统，实现用户点对点沟通、平台消息推送
5. **后台管理模块**：一站式管理后台，包含交易数据统计、商品审核、用户账号管控、商品分类配置功能

## 04 总结与展望

### 一、项目整体总结

1. **平台功能体系完整落地**：搭建一套流程规范、功能齐全的校园二手交易平台，完整实现用户管理、商品发布、供需匹配、订单交易、评价互动、后台管理全业务闭环，满足校园闲置交易全部核心需求。

2. **完整实践软件工程标准化流程**：全程严格按照软件工程开发规范执行，完整走完需求分析、系统架构设计、代码编码、功能测试、项目部署全流程，形成标准化、可复用的Web项目开发流程体系。

3. **深度落地主流技术栈，攻克多项技术难点**：熟练应用Spring Boot、React、MySQL等主流前后端开发技术，解决前后端接口联调、JWT身份认证、数据库复杂逻辑、订单状态机控制等核心技术难点；团队整体技术实操能力、标准化工程开发素养得到显著提升。

### 二、系统未来优化与扩展展望

1. **移动端适配优化**：开发移动端适配响应式页面，或独立开发小程序/APP客户端，优化手机端操作体验，实现随时随地使用平台。

2. **集成线上支付功能**：对接第三方安全支付接口，搭建线上资金交易通道，完善线上付款流程，提升交易便捷性与资金安全性。

3. **新增商品智能推荐引擎**：基于用户浏览记录、历史交易行为进行数据分析，实现个性化商品精准推送，进一步提升供需匹配效率。

4. **拓展平台社交生态**：增加用户关注、商品分享、商品评论互动等社交功能，提升平台用户活跃度与用户粘性。
