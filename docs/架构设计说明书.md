## 一、项目概述

### 1.1 项目背景

本系统面向高校学生群体，提供校园内闲置物品（教材、电子产品、生活用品等）的发布、浏览、交易与管理功能。系统采用 **B/S 架构**、**前后端分离** 设计，后端提供 RESTful API，前端以 React 单页应用（SPA）呈现图形化操作界面。

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| 前后端分离 | 前端与后端独立开发、独立部署，通过 HTTP/JSON 通信 |
| 可维护性 | 后端分层架构，前端按页面/组件模块化组织 |
| 数据规范 | 关系数据库满足第三范式（3NF），完整性约束完备 |
| 安全性 | JWT 无状态鉴权、密码 BCrypt 加密、权限分级控制 |
| 可扩展性 | 模块化设计，便于后续增加支付、通知推送等功能 |

## 四、后端架构设计

### 4.1 分层架构

```
请求 → JwtAuthFilter → Controller → Service → Mapper → MySQL
                              ↓
                        GlobalExceptionHandler
                              ↓
                         Result<T> 统一响应
```

| 层次 | 职责 | 关键类 |
|------|------|--------|
| **Controller** | 接收 HTTP 请求、参数校验、调用 Service、返回 JSON | `AuthController`, `GoodsController` 等 |
| **Service** | 业务逻辑、事务控制、权限校验 | `AuthService`, `OrderService` 等 |
| **Mapper** | 数据访问，MyBatis 接口 + XML | `UserMapper`, `GoodsMapper` 等 |
| **Entity** | 与数据库表对应的 Java 对象 | `User`, `Goods`, `TradeOrder` 等 |
| **DTO** | 前后端交互的请求体，带校验注解 | `RegisterRequest`, `GoodsRequest` 等 |
| **Config** | 安全、跨域、JWT、数据初始化 | `SecurityConfig`, `JwtAuthFilter` |
| **Common** | 统一响应封装、业务异常、全局异常处理 | `Result`, `BusinessException` |

### 4.2 安全架构

#### 认证流程

```
1. 用户 POST /api/auth/login → 后端校验账号密码（BCrypt）
2. 校验通过 → 生成 JWT（含 userId、role，有效期 24h）
3. 前端存储 Token 至 localStorage
4. 后续请求 Header 携带：Authorization: Bearer <token>
5. JwtAuthFilter 解析 Token → 写入 SecurityContext
6. Controller 通过 SecurityUtils.currentUserId() 获取当前用户
```

#### 权限控制

| 接口类型 | 策略 |
|----------|------|
| 公开接口 | `/api/auth/**`、商品浏览 `/api/goods`、分类列表 |
| 登录用户 | 发布商品、下单、消息、地址等 |
| 管理员 | `/api/admin/**`，Service 层校验 `role == 1` |

#### 密码安全

- 存储：`BCryptPasswordEncoder` 单向哈希
- 传输：HTTPS（生产环境建议启用）

### 4.3 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

| code | 含义 |
|------|------|
| 200 | 成功 |
| 400 | 业务错误（参数不合法、重复注册等） |
| 401 | 未登录 |
| 403 | 无权限 |
| 500 | 服务器内部错误 |

### 4.4 核心业务：订单状态机

```
                    ┌──────────┐
                    │ 0 待确认  │
                    └────┬─────┘
           卖家确认 ↓         ↓ 取消
              ┌──────────┐   ┌──────────┐
              │ 1 待交易  │   │ 3 已取消  │
              └────┬─────┘   └──────────┘
           双方完成 ↓
              ┌──────────┐
              │ 2 已完成  │ → 买家可评价 → 更新卖家信誉分
              └──────────┘
                    ↓
              商品状态 → 已售(2)
              调用存储过程 sp_complete_order
```

### 4.5 商品状态流转

| 状态值 | 含义 | 触发条件 |
|--------|------|----------|
| 0 | 待审核 | 用户发布商品 |
| 1 | 在售 | 管理员审核通过 |
| 2 | 已售 | 订单完成 |
| 3 | 已下架/驳回 | 卖家下架或管理员驳回 |

### 5.3 状态管理

本系统采用 **轻量级本地状态** 方案，未引入 Redux：

| 数据类型 | 存储方式 |
|----------|----------|
| 登录态（Token、userId、nickname、role） | `localStorage` |
| 页面数据（列表、表单） | React `useState` + `useEffect` |
| 全局 UI | Ant Design `ConfigProvider` |

工具函数（`api/index.js`）：

```javascript
getAuth()    // 读取 localStorage 登录信息
setAuth()    // 登录成功后写入
clearAuth()  // 退出登录清除
```

## 六、数据库架构设计

### 6.1 E-R 关系

```
sys_user ──1:N──→ goods ──1:N──→ trade_order ──1:1──→ order_review
    │                ↑                    │
    │                │                    │
    └──1:N──→ user_address          buyer/seller
    │
    └──1:N──→ user_message ←──N:1── sys_user
    
goods_category ──1:N──→ goods
```

### 6.2 数据表清单（7 张，满足 3NF）

| 表名 | 说明 | 主键 | 主要外键 |
|------|------|------|----------|
| `sys_user` | 用户 | user_id | — |
| `goods_category` | 商品分类 | category_id | — |
| `goods` | 商品 | goods_id | user_id, category_id |
| `trade_order` | 交易订单 | order_id | goods_id, buyer_id, seller_id |
| `order_review` | 订单评价 | review_id | order_id, reviewer_id |
| `user_message` | 站内消息 | message_id | sender_id, receiver_id, goods_id |
| `user_address` | 收货地址 | address_id | user_id |

### 6.3 完整性约束

| 类型 | 示例 |
|------|------|
| 主键 | 每表 AUTO_INCREMENT 主键 |
| 唯一 | 学号、邮箱、订单号、每订单一条评价 |
| 外键 | goods→user, order→goods/user |
| CHECK | `price > 0`，`rating BETWEEN 1 AND 5` |
| 非空 | 标题、密码、昵称等核心字段 |

### 6.4 存储过程

**`sp_complete_order(p_order_id)`** — 订单完成时：

1. 校验订单状态为「待交易(1)」
2. 更新订单 → 已完成(2)，记录完成时间
3. 更新关联商品 → 已售(2)
4. 重算卖家平均评分，更新信誉分

### 6.5 触发器

**`trg_order_status_before_update`** — 禁止已完成订单被修改状态：

```sql
IF OLD.status = 2 AND NEW.status != 2 THEN
  SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '已完成订单不可变更状态';
END IF;
```

## 八、关键业务流程

### 8.1 完整交易流程

```
买家                    系统                     卖家
 │                       │                       │
 │    浏览商品列表        │                       │
 │──────────────────────→│                       │
 │    查看商品详情        │                       │
 │──────────────────────→│                       │
 │    创建订单            │                       │
 │──────────────────────→│  通知卖家（消息）      │
 │                       │──────────────────────→│
 │                       │     卖家确认订单       │
 │                       │←──────────────────────│
 │                       │     线下/邮寄交易      │
 │                       │←──────────────────────│
 │    双方确认完成        │                       │
 │──────────────────────→│  sp_complete_order    │
 │                       │  商品→已售            │
 │    买家评价            │  更新卖家信誉         │
 │──────────────────────→│                       │
```

### 8.2 商品发布审核流程

```
卖家发布 → status=0(待审核) → 管理员审核
                                  ├── 通过 → status=1(在售)
                                  └── 驳回 → status=3(下架)
```
