## 03 已完成功能模块详解

### 一、用户子系统：JWT 无状态鉴权

**核心流程**：
```
用户输入账号密码 → 后端 BCrypt 校验密码 → 生成 JWT（含 userId + role + 昵称，有效期24h）
    → 前端存储 Token 到 localStorage → 后续请求 Header 携带 Authorization: Bearer <token>
    → JwtAuthFilter 解析 Token → 写入 SecurityContext → Service 层通过 SecurityUtils 获取当前用户
```

**关键代码要点**：

- **密码加密**：使用 Spring Security 自带 `BCryptPasswordEncoder`，注册时加密存储，登录时哈希比对
- **JWT 生成** (`JwtUtil.java`)：使用 `jjwt` 库，Payload 包含 `userId`, `nickname`, `role`, `exp`
- **权限分级**：
  - `role = 0` → 普通学生用户（可浏览、发布、下单、发消息）
  - `role = 1` → 管理员（额外可访问 `/api/admin/**`）
- **路由守卫** (`PrivateRoute.jsx`)：前端检查 localStorage 是否有 Token，无则跳 `/login`；管理员页额外校验 `role`

### 二、商品子系统：发布 → 审核 → 展示 → 搜索

**商品状态流转**：
```
用户发布 → status=0(待审核) → 管理员审核
                           ├── 通过 → status=1(在售) → 前台展示
                           └── 驳回 → status=3(已下架) → 不可展示
      在售商品 → 卖家手动下架 → status=3
      商品被下单完成 → status=2(已售)（由存储过程自动更新）
```

**核心功能**：
- **发布**：标题、描述、价格、原价（可选）、成色、多图片、交易地点
- **浏览**：首页卡片列表，展示图片+标题+价格+卖家昵称，支持分页
- **搜索**：关键词模糊搜索 + 分类筛选
- **详情**：商品详情页展示完整信息、卖家信息，支持"立即下单"、"联系卖家"
- **我的商品**：卖家查看自己发布的商品列表，支持下架操作

### 三、订单子系统：四状态状态机

**订单状态流转图**：
```
                   ┌────────────┐
                   │ 0 待确认   │  ← 买家下单后初始状态
                   └─────┬──────┘
          卖家确认 ↓         ↓ 买家/卖家取消
              ┌────────────┐   ┌────────────┐
              │ 1 待交易   │   │ 3 已取消   │
              └─────┬──────┘   └────────────┘
          双方完成交易 ↓
              ┌────────────┐
              │ 2 已完成   │  → 自动调用存储过程 sp_complete_order
              └────────────┘       → 商品 status=2(已售) + 完成时间记录
```

**业务校验** (`OrderService.java`)：
- 卖家不能购买自己的商品
- 只有卖家可确认订单
- 只有订单参与者（买家/卖家）可完成/取消
- 已完成订单不可修改状态（数据库触发器强制约束）
- 邮寄方式必须填写收货地址

### 四、评价与信誉分：闭环信用体系

**评价规则**：
- 仅买家可评价，且订单必须处于"已完成"状态
- 单个订单仅允许一条评价（数据库 UNIQUE 约束）
- 评分限制 1-5 星（数据库 CHECK 约束）

**信誉分自动更新**：
- 方案一：调用存储过程 `sp_complete_order` 时，同时根据已有评价计算平均分
- 方案二：买家提交评价后，Service 层通过 `reviewMapper.avgRatingBySeller()` 查询历史平均评分，更新 `sys_user.credit_score`

### 五、站内消息子系统：点对点即时沟通

**消息结构**：
- `sender_id`：发送者
- `receiver_id`：接收者
- `goods_id`：关联商品（可选，便于上下文）
- `content`：消息内容（TEXT 类型，支持长文本）
- `is_read`：已读状态
- `create_time`：时间戳

**前端展示**：
- 左侧会话列表（按对话对方分组，最近消息优先）
- 右侧聊天记录展示（气泡样式，区分自己/对方消息）
- 底部输入框发送消息

### 六、地址子系统：多收货地址管理

- 支持地址新增、修改、删除
- 支持设置默认地址（下单时自动填充）
- 字段：收件人、手机号、详细地址、是否默认

### 七、管理员后台：一站式运营管理

**管理员功能**：
1. **数据统计 Dashboard**：用户总数、商品总数、订单总数、在售商品数、待审核商品数
2. **用户管理**：用户列表，启用/禁用账号（`status` 字段控制）
3. **商品审核**：待审核商品列表，一键通过/驳回
4. **分类管理**：商品分类 CRUD，支持调整排序

### 八、文件上传：商品图片本地存储

- 上传路径：后端 `uploads/` 目录（由 `WebConfig.java` 配置静态资源映射）
- 访问路径：`http://localhost:8080/uploads/文件名`
- 限制：单文件 ≤ 5MB，请求总大小 ≤ 10MB
- 图片字段：`goods.images` 使用 JSON 格式存储，支持多张图片

## 04 后端核心实现

### 一、统一响应格式 `Result<T>`

```java
// Result.java - 规范前后端数据格式
{
  "code": 200,        // 200成功, 400业务错误, 401未登录, 403无权限, 500服务器错误
  "message": "success", // 提示信息
  "data": { ... }      // 返回数据（任意类型 T）
}
```

**全局异常处理**：`@RestControllerAdvice` 统一捕获 `BusinessException` 和其他异常，确保异常也以统一 JSON 格式返回。

### 二、JWT 认证流程详解

**登录接口** (`AuthController.java`)：
```java
@PostMapping("/login")
public Result<LoginResponse> login(@Valid @RequestBody LoginRequest request) {
    return Result.ok(authService.login(request));
}
```

**登录服务**内部流程：
1. 根据学号查用户 → `userMapper.findByStudentNo(studentNo)`
2. BCrypt 校验密码 → `passwordEncoder.matches(raw, encoded)`
3. 校验账号是否被禁用
4. 生成 JWT Token（有效期 24 小时）→ `jwtUtil.generateToken(user)`
5. 返回 Token + 用户基本信息

**JwtAuthFilter 过滤器**：
- 每个请求到达 Controller 前执行
- 从 `Authorization: Bearer xxx` 中提取 Token
- 调用 `jwtUtil.validateToken()` 校验签名和有效期
- 解析出 `userId`, `nickname`, `role` 写入 `SecurityContext`
- Controller 层通过 `SecurityUtils.currentUserId()` 获取当前用户

### 三、订单完成：存储过程调用

`OrderService.completeOrder()` 中关键代码：
```java
jdbcTemplate.update("CALL sp_complete_order(?)", orderId);
```

这体现了**应用层 + 数据库层协同**的设计思想：复杂多表更新由数据库存储过程原子执行，保证数据一致性，同时减少应用层代码冗余。

### 四、事务控制

在 `OrderService` 的 `createOrder`、`completeOrder`、`reviewOrder` 等方法上标注 `@Transactional`，保证多步数据库操作的原子性——任何一步失败，全部回滚。

## 06 数据库设计与高级特性

### 一、7 张核心数据表

| 表名 | 说明 | 核心字段 | 约束 |
|------|------|---------|------|
| `sys_user` | 用户表 | user_id, student_no(UNIQUE), email(UNIQUE), password, nickname, avatar, phone, college, credit_score, role, status | 学号/邮箱唯一 |
| `goods_category` | 商品分类 | category_id, name(UNIQUE), sort_order, status | 分类名唯一 |
| `goods` | 商品表 | goods_id, user_id(FK), category_id(FK), title, description, price, original_price, condition_level, images(JSON), trade_location, view_count, status | price > 0 CHECK |
| `trade_order` | 订单表 | order_id, order_no(UNIQUE), goods_id(FK), buyer_id(FK), seller_id(FK), amount, delivery_type, address, status, create_time, finish_time | 订单号唯一 |
| `order_review` | 评价表 | review_id, order_id(UNIQUE, FK), reviewer_id(FK), rating, content, create_time | rating 1-5 CHECK，订单唯一评价 |
| `user_message` | 消息表 | message_id, sender_id(FK), receiver_id(FK), goods_id(FK), content, is_read, create_time | — |
| `user_address` | 地址表 | address_id, user_id(FK), receiver, phone, detail, is_default | — |

**范式说明**：7 张表满足第三范式（3NF），无冗余数据，每列依赖主键。

### 二、存储过程 `sp_complete_order`

**功能**：完成订单时，一次性执行三项操作：
1. 将订单状态更新为「已完成(2)」，记录 `finish_time = NOW()`
2. 将关联商品状态更新为「已售(2)」
3. 重新计算卖家平均评分并更新 `sys_user.credit_score`

**优点**：
- **原子性**：整个操作在数据库一个事务中完成，不会出现中间状态
- **性能**：减少应用与数据库往返次数
- **可维护性**：业务逻辑集中在数据库层，便于 DBA 优化调整

### 三、触发器 `trg_order_status_before_update`

**触发时机**：订单表 `trade_order` 执行 UPDATE 前

**业务规则**：`IF OLD.status = 2 AND NEW.status != 2 THEN 抛出异常`

**目的**：从数据库底层强制防止已完成订单被修改状态，即便应用层存在 Bug，数据库层也有最后一道防线。这体现了"**防御性编程 + 数据完整性约束**"的设计思想。
